§01 · Purpose

Text entry with four visual states.

LSTextField and LSTextArea are the only typed text entry atoms in the Copper system. Both expose the same four states: default, focused, error, disabled. Both share structural tokens: 48px height (input height), 6px radius (radius.sm), 12px padding (spacing.3).

The focused state resolves the border to color.border.focus (copper-500, #EE7C2B) with a 3px glow ring. This is the single most important affordance in the system — it tells the rider exactly where their keyboard input is going in a high-vibration, gloved context.

§03 · States / Interactions

Focus, keyboard, accessibility.

The focused state shows the copper border + glow — not a filled background. The field stays the same color as default (surface.card) so text on the copper border has sufficient contrast. WCAG 2.1 AA requires 3:1 contrast between the focus indicator and adjacent non-focused elements. The 3px copper ring at 15% opacity fulfills this combined with the solid 1px copper border.

LSTextArea grows vertically as the user types. In production iOS this is a UITextView with intrinsic content size updates on every keystroke. On Android, a Compose TextField with minLines/maxLines. Both platforms must not animate the height growth — it should be instantaneous to avoid fighting the keyboard animation.

Error state is set externally by the form owner — the atom does not validate itself. The helper text row exists below both field types and must expand without truncation. Error text uses content.error (#C9423C), not an inline red literal.

§05 · Motion

Border transitions, no height animation.

Border-color and box-shadow transitions use 120ms ease. This is slightly faster than button transitions (100ms) because focus changes must not lag behind the keyboard appearance.

TextArea height growth is instantaneous — no CSS height animation. Animating height during keyboard appearance creates a double-animation fight on mobile that feels broken. The content reflow is the affordance.

§06 · Rationale

Why 48px for inputs?

Acceptance Criteria Coverage

  • AC-1 — LSTextField default: surface.input bg, border.default stroke, radius.sm, spacing.3 pad.
  • AC-2 — Android identical values confirmed in story parity (Compose TextField).
  • AC-3 — Focused state: border.focus (#EE7C2B) with 3px copper glow ring.
  • AC-4 — Error state: border.error + helper text in content.error.
  • AC-5 — Leading icon slot (.search) renders at sizing.icon.sm (16px) with spacing.2 gap.
  • AC-6 — Sandbox stories cover all 4 states per component on both platforms.
  • AC-7 — Live args.value control reflects typed text in real time.

Design Rationale

48px input height exceeds the 44px button height intentionally. Inputs are stationary tap targets — the user aims at a labeled field and taps once. The extra 4px improves hit accuracy for gloved or distracted riders without visually overwhelming the form.

radius.sm (6px) is softer than radius.md (10px for buttons) but harder than a pill. It reads as utilitarian — a form field, not a decorative chip. The copper focus ring is the system's primary interactive affordance and must not be compromised by making it too subtle.
