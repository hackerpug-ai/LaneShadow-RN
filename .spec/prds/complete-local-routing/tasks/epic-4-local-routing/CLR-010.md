## CRITICAL CONSTRAINTS

1. **MUST** calculate routes without internet connection using downloaded offline data - core offline requirement
2. **STRICTLY** validate waypoints are within downloaded offline regions - prevent routing failures
3. **MUST** return provider-agnostic route geometry format - enable Convex storage without vendor lock-in
4. **NEVER** exceed 2-second calculation time for typical routes (under 50 miles) - performance requirement
5. **MUST** handle offline data missing gracefully with clear error messaging - user experience requirement

---

## SPECIFICATION

**Objective:** Implement offline route calculation using Mapbox SDK with waypoint support, error handling for insufficient offline data, and provider-agnostic geometry storage.

**Success State:** Route calculation works completely offline using downloaded map data, supports up to 10 waypoints, validates region coverage, returns standardized geometry format, and completes in under 2 seconds for typical routes.

## DESIGN

**References:**
- @rnmapbox/maps Directions API: https://github.com/rnmapbox/maps/docs/directions.md
- PRD Phase 2: Offline Maps & Local Routing - "On-Device Route Calculation" section
- Mapbox Directions Documentation: https://docs.mapbox.com/api/navigation/directions/

**Patterns:**
- Async/await for route calculation
- Validation before calculation
- Provider-agnostic data structures
- Error handling with fallback options
- Result caching for performance

**Anti-Patterns:**
- **DO NOT** make network requests during offline calculation
- **DO NOT** skip offline region validation (will cause failures)
- **DO NOT** store Mapbox-specific data in Convex
- **DO NOT** block UI thread during calculation
- **DO NOT** ignore waypoint order

**Route Calculation Interface:**
```typescript
interface RouteCalculation {
  calculateRoute(params: RouteParams): Promise<RouteResult>
  validateOfflineCoverage(waypoints: Waypoint[]): Promise<CoverageResult>
  estimateTime(params: RouteParams): number
}

interface RouteParams {
  waypoints: Waypoint[]  // Array of {lat, lng} coordinates
  profile: 'driving' | 'cycling' | 'walking'
  optimize: boolean  // Reorder waypoints for optimal route
}

interface RouteResult {
  geometry: string  // Encoded polyline (Mapbox format)
  bounds: {
    northeast: { lat: number; lng: number }
    southwest: { lat: number; lng: number }
  }
  distance: number  // Meters
  duration: number  // Seconds
  legs: RouteLeg[]  // Individual segments between waypoints
}
```

**Offline Validation Flow:**
```
Request Route → Validate Offline Coverage
  ├─ Coverage Complete → Calculate Offline
  ├─ Coverage Partial → Suggest Downloads
  └─ No Coverage → Offer Online Fallback
```

## NOTES

**Waypoint Limit:**
- Maximum 10 waypoints per route (Mapbox limitation)
- Optimize waypoint order for multi-stop routes
- Preserve user-specified order when optimization disabled

**Offline Region Requirements:**
- All waypoints must be within downloaded regions
- Route path must stay within downloaded regions
- Download larger regions to avoid coverage gaps
- Validate coverage before calculation to prevent failures

**Coordinate System:**
- Input: [lat, lng] format (Google-compatible)
- Output: Encoded polyline (Mapbox format)
- Conversion handled by geometry encoder

**Error Scenarios:**
- No offline data: Suggest region download
- Partial coverage: Suggest additional regions
- Corrupted data: Offer re-download
- Network timeout: Retry with exponential backoff

**Performance Optimization:**
- Cache route results for identical requests
- Use async/await to avoid blocking UI
- Show loading indicator during calculation
- Simplify geometry for long routes
