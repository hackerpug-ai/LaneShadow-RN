### AC-001: Model Download Failure Handling
**GIVEN** model download fails due to network error
**WHEN** download failure occurs
**THEN** system displays user-friendly error message
**AND** system provides "Retry Download" button
**AND** system logs technical error details
**AND** system offers "Contact Support" after 3 failed retries

**Verify:** Simulate network failure during download, verify error message appears, test retry functionality.

### AC-002: Corrupted Model Cache Recovery
**GIVEN** model cache file is corrupted
**WHEN** app validates model on launch
**THEN** system detects corruption via checksum validation
**AND** system displays "Your Shadow needs to be restored" message
**AND** system deletes corrupted cache automatically
**AND** system initiates re-download without user intervention

**Verify:** Corrupt model file, launch app, verify error message and automatic re-download.

### AC-003: Out-of-Disk-Space Handling
**GIVEN** device has insufficient storage for model download
**WHEN** user attempts to download model
**THEN** system checks available storage before download
**AND** system displays "Not Enough Storage" error with required amount
**AND** system provides "Free Up Space" guidance
**AND** system blocks download until space is available

**Verify:** Fill device storage, attempt download, verify error message with storage requirements.

### AC-004: Concurrent Inference Request Queuing
**GIVEN** multiple enrichment requests are triggered rapidly
**WHEN** inference requests overlap
**THEN** system queues requests to prevent model overload
**AND** system processes requests sequentially with priority
**AND** system provides queue status to user
**AND** system cancels stale requests automatically

**Verify:** Trigger multiple route creations rapidly, verify requests queue properly, check no model crashes.

### AC-005: Offline Data Missing Error Handling
**GIVEN** user requests route calculation without offline data
**WHEN** offline regions are not downloaded
**THEN** system displays "Offline Data Required" error
**AND** system suggests which regions to download
**AND** system provides "Download Regions" button
**AND** system falls back to online routing if user opts in

**Verify:** Request route outside downloaded regions, verify error message with download suggestions.

### AC-006: Weather API Failure Fallback
**GIVEN** weather data fetch fails
**WHEN** enrichment attempts to load weather
**THEN** system logs weather API error
**AND** system continues enrichment without weather data
**AND** system displays partial enrichment (labels without weather)
**AND** system retries weather fetch in background

**Verify:** Simulate weather API failure, verify enrichment continues without weather, check background retry.

## DESIGN

**References:**
- React Error Boundaries: https://react.dev/reference/react/Component#catching-rendering-errors-with-an-error-boundary
- PRD Phase 7: Hybrid Enrichment Polish - "Edge Case Handling" section
- Error Handling Best Practices: User-friendly messages + technical logs

**Patterns:**
- Error boundary pattern for React components
- Try-catch with user-friendly messages
- Retry logic with exponential backoff
- Fallback behavior for degraded experience
- Comprehensive logging for debugging

**Anti-Patterns:**
- **DO NOT** show raw error objects to users
- **DO NOT** crash app on edge cases
- **DO NOT** skip error logging (hard to debug)
- **DO NOT** leave users without recovery options
- **DO NOT** use technical jargon in error messages

**Error Handler Structure:**
```typescript
interface ErrorHandler {
  handle(error: Error): UserMessage
  recover(error: Error): RecoveryAction
  log(error: Error): void
}

interface UserMessage {
  title: string
  message: string
  action?: RecoveryAction
}

interface RecoveryAction {
  label: string
  handler: () => void
}
```

**Error Message Mapping:**
```typescript
const errorMessages: Record<string, UserMessage> = {
  'MODEL_DOWNLOAD_FAILED': {
    title: 'Download Failed',
    message: 'Your Shadow download was interrupted. Please check your connection and try again.',
    action: { label: 'Retry', handler: retryDownload }
  },
  'MODEL_CORRUPTED': {
    title: 'Shadow Corrupted',
    message: 'Your Shadow needs to be restored. We\'ll download it again automatically.',
    action: { label: 'Restore', handler: redownloadModel }
  },
  'OUT_OF_SPACE': {
    title: 'Not Enough Storage',
    message: `Your Shadow needs 800MB of free space. Please free up some space and try again.`,
    action: { label: 'Open Settings', handler: openStorageSettings }
  }
}
```

## NOTES

**Error Message Principles:**
- Use plain language (no technical jargon)
- Explain what happened (clear cause)
- Provide next steps (recovery action)
- Offer help option (contact support)
- Maintain brand voice (copper-accented, industrial-warm)

**Recovery Strategies:**
- **Retry:** For transient failures (network, timeout)
- **Restore:** For corrupted data (re-download)
- **Fallback:** For missing features (partial enrichment)
- **Guidance:** For user action required (free storage)
- **Escalate:** For unrecoverable errors (contact support)

**Logging Requirements:**
- Log technical error details (stack trace, error code)
- Include user context (route ID, action, timestamp)
- Sanitize sensitive data (no tokens, personal info)
- Use structured logging (JSON format)
- Send to error tracking service (e.g., Sentry)

**Testing Edge Cases:**
- Use network link conditioner for iOS
- Use Chrome DevTools for Android network throttling
- Simulate low storage scenarios
- Test concurrent actions (rapid tapping)
- Verify error boundary catches React errors
