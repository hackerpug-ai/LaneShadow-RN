## CRITICAL CONSTRAINTS

1. **MUST** persist download state to Convex immediately on every progress update (minimum 5% increments)
2. **NEVER** store model bytes in Convex — only metadata and progress percentage
3. **STRICTLY** use atomic writes for local file storage — prevent partial file corruption on crash
4. **MUST** validate stored progress against actual file size on app resume — detect and repair inconsistencies
5. **NEVER** allow progress to regress — if stored progress > actual file size, trust file size over stored value

---

## SPECIFICATION

Implement persistent download state management that survives app termination, device restarts, and network interruptions, ensuring model download resumes from last known good state without data loss or corruption. Success state: Download progress persists across app restarts, resumes correctly from interrupted state, and never requires full re-download unless file is corrupted.

---

## ACCEPTANCE CRITERIA

### AC-001: Happy Path — Progress Persists Across App Restarts
**GIVEN** model download is at 47% complete
**WHEN** user force-quits app and relaunches
**THEN** download screen shows 47% complete immediately AND download resumes from 47% WITHOUT re-downloading first 47%

### AC-002: Edge Case — Device Restart During Download
**GIVEN** model download is at 82% complete
**WHEN** device is restarted (power cycle)
**THEN** after app launch, download shows 82% complete AND resumes from 82% AND completes remaining 18% successfully

### AC-003: Edge Case — Stored Progress Mismatch Recovery
**GIVEN** stored progress shows 60% but actual file is only 55% complete (crash during write)
**WHEN** app launches and validates file size
**THEN** progress is corrected to 55% AND download resumes from 55% AND stored progress is updated to match reality

### AC-004: Error Case — Corrupted Partial File Detection
**GIVEN** download was interrupted at 30% and file bytes are corrupted
**WHEN** app launches and validates checksum of partial file
**THEN** corrupted file is deleted AND stored progress is reset to 0% AND download starts from beginning

## DESIGN

### Schema Design
```typescript
// convex/schema.ts
const modelDownloadProgress = defineTable({
  userId: v.string(),
  progressPercent: v.number(), // 0-100
  bytesDownloaded: v.number(),
  totalBytes: v.number(),
  lastUpdate: v.number(), // Timestamp
  checksum: v.optional(v.string()), // SHA-256 when complete
  version: v.string(), // Model version (e.g., "qwen3.5-0.8b-v1")
}).index("by_user", ["userId"]);
```

### State Machine
```
NOT_STARTED → DOWNLOADING (0-99%) → COMPLETE (100%)
     ↓              ↓
   [INTERRUPTED] → RESUME → DOWNLOADING
     ↓
   [CORRUPTED] → DELETE → NOT_STARTED
```

### Atomic Write Pattern
```typescript
// lib/ai/atomic-write.ts
async function atomicWrite(path: string, data: Buffer): Promise<void> {
  const tempPath = `${path}.tmp`;
  await fs.writeFile(tempPath, data);
  await fs.rename(tempPath, path); // Atomic on POSIX
}
```

### Validation Logic
```
On App Resume:
1. Read stored progress from Convex
2. Check actual file size on disk
3. If |stored - actual| > 1% of total: Correct stored progress
4. Validate checksum of existing bytes
5. If checksum invalid: Delete file, reset to 0%
6. Resume download from actual bytes
```
