# Complete On-Device Routing for LaneShadow - PRD

**Epic ID:** TBD
**Status:** In progress — Epics 1–4 complete; v1.3 rollback of on-device LLM features in progress
**Last Updated:** 2026-04-10

Implement complete offline routing capability for LaneShadow's AI-native motorcycle ride planner by migrating from Google Maps to Mapbox with @rnmapbox/maps and @trestleinc/replicate for local-first sync, eliminating recurring API costs while preserving copper-accented dark theme and weather overlays. All route enrichments (weather, leg labels, AI descriptions) load progressively and asynchronously on the server, returning routes immediately after local routing completes. Routes are editable offline with automatic conflict resolution via Yjs CRDTs.

**Product Context:** LaneShadow is an AI-native motorcycle ride planner — map-first, conversation-driven. Tagline: "Ride the Moment" — turn a feeling into a road. Platforms: iOS and Android (React Native + Expo). Aesthetic: Rugged, industrial-warm, copper-accented, dark-first.

> ## ⚠️ v1.3 On-Device LLM Rollback (2026-04-10)
>
> The 2026-04-10 environment-bias research finding (`.spec/research/local-models/ENVIRONMENT_BIAS_FINDING_2026-04-10.md`) established that all prior Qwen3.5 0.8B latency measurements were taken on a 2026 MacBook Pro running MLX — a Mac-only runtime that does not reflect real mobile hardware. Estimated iPhone 15/16 Pro latency for the same model is ~6–15s per inference, which is unacceptable for interactive route planning.
>
> **As of v1.3, this PRD removes all on-device LLM functionality from scope.** Specifically:
>
> - **No Qwen3.5 0.8B anywhere.** No "Download Your Shadow" onboarding. No Phase 0 Shadow Setup. No `mlx-local` dependency. No Core ML bundle. No ONNX bundle. No model file in the app bundle. No local model cache. No local inference path.
> - **Leg labels are deterministic, derived from waypoint names** (e.g., `"San Francisco → Daly City"`) generated by pure code at route-creation time. They are available instantly, offline, and require no model.
> - **Creative enrichment (route label, rationale, highlights) is Haiku-only, server-side, progressive.** It runs when the device is online and merges into the route document asynchronously. Offline routes remain usable indefinitely with their deterministic leg labels; creative enrichment lands the next time the device connects.
> - **Epics 1–4 are marked complete** in the task tracker, but any code they introduced for local model download / setup / runtime / inference is subject to rollback. See the "Rollback Scope" section below.
>
> The non-LLM parts of this PRD — Mapbox SDK, offline map regions, `@trestleinc/replicate` sync, weather overlays, progressive weather enrichment, Haiku-only creative enrichment, mini-maps — are **unchanged** by v1.3. Everything that wasn't an on-device LLM feature still ships as planned.

## Background

### Current State

- **Map Provider:** Google Maps API
- **Routing:** Server-side via Google Directions API
- **Cost:** ~$1,500/month at 1,000 routes/day
- **Dependencies:** Requires internet connection for routing
- **Weather Overlays:** Custom polyline rendering system
- **Mini-maps:** Route attachment cards with overview geometry

### Problem Statement

1. **Cost:** Google Maps pricing becomes prohibitive at scale
2. **Offline Limitations:** Cannot calculate routes without internet
3. **Vendor Lock-in:** Tightly coupled to Google's polyline format and API
4. **Reliability:** Service dependency on Google's infrastructure

### Solution Overview

Migrate to Mapbox SDK with @rnmapbox/maps for offline routing, and implement progressive server-side enrichment (weather + Haiku creative labels) that merges into the route document as it arrives. Maintain all UI/UX while reducing costs by ~98%.

**Architecture:**
- **Local (instant):** Route editing with automatic conflict resolution via @trestleinc/replicate (Yjs CRDTs + op-sqlite)
- **Local (<10ms):** Deterministic leg labels derived from waypoint names (e.g., `"San Francisco → Daly City"`) at route-creation time. Pure code, no model, works offline.
- **Local (offline):** Mapbox SDK calculates route geometry from downloaded map regions
- **Cloud (<20s):** Weather data fetched asynchronously by a Convex background job and merged progressively into the route document
- **Cloud (~3.9s):** Haiku generates creative enrichment (label, rationale, highlights) on the server as a background job; results merge into the route document when they arrive
- **Progressive Enhancement:** Routes display immediately with geometry + deterministic leg labels; weather badges and creative text fade in as each enrichment phase completes
- **Offline-First Sync:** Bidirectional CRDT delta sync when online; offline edits are preserved and merged automatically

**What's explicitly NOT in this architecture:**
- No on-device language model (no Qwen3.5, no `mlx-local`, no Core ML bundle, no ONNX)
- No "Download Your Shadow" onboarding or model download manager
- No local model cache, no local inference code path
- No client-side LLM dependency of any kind

## Technical Approach

### Architecture Decision: Mapbox SDK (no on-device LLM)

**Selected Approach:** Mapbox + @rnmapbox/maps + @trestleinc/replicate + Haiku-only server-side enrichment

**Rationale:**
- ✅ Preserves current React Native + Expo architecture
- ✅ Maintains weather overlay system (polyline rendering works identically)
- ✅ True offline route geometry without native modules
- ✅ Proven technology (BMW, The Weather Channel, etc.)
- ✅ Reasonable implementation effort (2-3 months vs 6+ months for a custom engine)
- ✅ First-class React Native support via op-sqlite (no WASM hacks required)
- ✅ Automatic conflict resolution via Yjs CRDTs
- ✅ Zero on-device ML runtime — no model downloads, no memory pressure, no mobile-latency unknowns

**Rejected Alternative:** Custom routing engine with native bridge
- Requires iOS/Android native developers (2)
- C++ knowledge for OSRM/Valhalla integration
- 6+ month timeline
- Only justified for regulatory requirements preventing third-party services

### Technology Stack

```typescript

### Phase 2: Offline Maps & Local Routing (Weeks 3-5)

**Goal:** Download map regions and calculate routes offline

#### Tasks

1. **Offline Region Download**
   - Implement `lib/mapbox/offline-manager.ts`
   - Create UI for region selection (bounding box input)
   - Progress tracking for downloads
   - Storage management (list, delete downloaded regions)

2. **On-Device Route Calculation**
   - Implement `lib/mapbox/routing.ts`
   - Offline Directions API using cached routing data
   - Support for waypoints and multi-stop routes
   - Error handling for insufficient offline data

3. **Deterministic Leg Label Derivation**
   - Implement `lib/routing/leg-labels.ts`
   - Pure function: given an ordered list of waypoints with names, return `"FromName → ToName"` strings per leg
   - When a waypoint has no name (e.g., user-dropped pin), fall back to a reverse-geocode lookup when online, or to a coordinate-based placeholder offline
   - No model, no inference, no async work — available synchronously at route-creation time
   - Works identically online and offline

4. **Local-First Sync with Replicate**
   - Install `@trestleinc/replicate` and `@op-engineering/op-sqlite`
   - Install crypto polyfills (`react-native-get-random-values`, `react-native-random-uuid`)
   - Create `collections/use-routes.ts` with Replicate collection
   - Define route schema with `schema.define()` for versioning
   - Implement server-side collection with auth hooks
   - Test offline route editing (add waypoint, rename, reorder)
   - Verify bidirectional sync when online

5. **Convex Integration**
   - Install Replicate component in `convex/convex.config.ts`
   - Store route calculations with provider-agnostic format
   - Implement route plan mutations and queries

#### Success Criteria
- Can download map region for offline use
- Route calculation works without internet
- Downloaded regions persist across app restarts (managed by Mapbox SDK)
- Convex stores route geometry provider-agnostically
- Leg labels render instantly at route-creation time via deterministic waypoint-name derivation (no model, no async)
- **Route edits sync automatically via CRDT deltas**
- **Offline route editing works instantly (no network required)**
- **Conflict resolution merges concurrent edits automatically**

---

### Phase 3: Progressive Server-Side Enrichment (Week 6)

**Goal:** Orchestrate server-side creative enrichment with progressive enhancement

#### Tasks

1. **Server-Side Enrichment Orchestration**
   - Implement `convex/actions/enrichment/progressive.ts`
   - On route insert, schedule a weather-fetch background job (Open-Meteo)
   - On route insert, schedule a Haiku creative-enrichment background job (label, rationale, highlights)
   - State machine: `draft → weather-ready → complete`
   - Both background jobs merge results into the route's `route_enrichments` document as they arrive
   - Offline routes queue their enrichment jobs; jobs run when the device reconnects and the CRDT sync fires the `onInsert` hook

2. **Progressive Enhancement UI**
   - Show route geometry + deterministic leg labels immediately
   - Display skeleton states for weather badges and creative text during enrichment
   - Incremental UI updates as each enrichment phase arrives
   - Enrichment status badges (weather-ready → complete)

#### Success Criteria
- Route geometry + deterministic leg labels appear instantly (no model, no waiting)
- Server-side enrichment completes in background without blocking UI
- **CRDT sync completes automatically when online**
- UI updates reactively as each enrichment phase arrives
- Offline routes remain usable indefinitely; creative enrichment lands the next time the device connects
- No blocking failures (deterministic leg labels always available as the base state)

// convex/functions/routes.ts
export const saveRoute = mutation({
  args: {
    geometry: v.string(), // Encoded polyline (Mapbox format)

      northeast: v.object({ lat: v.number(), lng: v.number() }),
      southwest: v.object({ lat: v.number(), lng: v.number() }),
    }),
    metadata: v.any() // Weather overlays, etc.
  },
  handler: async (ctx, args) => {
    // Store route data
  }
})
```
