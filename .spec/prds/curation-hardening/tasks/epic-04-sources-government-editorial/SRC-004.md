--------------------------------------------------------------------------------

**Problem:** Epic 4 needs a geometric discovery source, but the repo does not yet have a curvature consumer module or a documented provenance handoff for the precomputed artifact.

**Why it matters:** Curvature discovery is the only surviving “hidden gem” source after the scope reduction. If this task reruns raw geometry processing or floods the catalog with already-known roads, it defeats the purpose.

**Current state:** There is no `curvature_discovery.py` source module. The PRD calls for consuming precomputed output, but the artifact location still needs to be anchored and documented. Existing UI/detail surfaces can likely show provenance, but source-specific rendering/tests do not yet exist.

**Desired state:** A deterministic consumer reads the precomputed artifact, excludes existing catalog matches, and emits twisties candidates with provenance that survives to the existing detail surface.

**Artifact contract:** Unless explicitly overridden, the consumer must read from the canonical cached artifact path `data/curvature/adamfranco-us-curvature.jsonl`. An operator may override that path with a CLI flag like `--artifact` or an environment variable `CURVATURE_ARTIFACT_PATH`. If none of those locations exist, the task must fail loudly rather than guess a source file.

**Storage follow-up plan:** The accepted production strategy is to use Convex File Storage for the raw curvature artifact release files, keep decomposed per-state JSONL shards in File Storage instead of Convex document tables, keep one metadata table for releases and one for shards with storage IDs as pointers, and keep all product/runtime reads on `curated_routes` rather than on the raw artifact payloads. That storage surface is now live on the dev deployment.
