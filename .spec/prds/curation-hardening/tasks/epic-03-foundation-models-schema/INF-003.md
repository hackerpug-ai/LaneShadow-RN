BACKGROUND
--------------------------------------------------------------------------------

**Problem:** The Convex schema needs (a) new optional semantic matching + enrichment fields on `curated_routes`, (b) a native `vectorIndex` on `searchEmbedding` so the pipeline can do fast cosine-similarity retrieval, (c) a new `route_posts_raw` table holding raw LLM extraction artifacts per community post, and (d) a new `route_matches` table holding the audit log of (post → route) match decisions. All schema changes must be non-breaking so existing mobile clients continue to work.

**Why it matters:** This is the backbone of the semantic matching pipeline. INF-004 writes embeddings into `searchEmbedding`. INF-006 queries the `by_embedding` vectorIndex. Sprint 9 writes raw LLM artifacts into `route_posts_raw`. Sprint 6/9/10 write match audit rows into `route_matches`. Without this schema, none of the downstream work can happen.

**Current state:** `convex/schema.ts` has `curated_routes` using `curatedRouteValidator` from `convex/models/curated-routes.ts`. Existing fields include name, state, source, centroid coords, 6 scores, archetype, content fields. Location (GeoJSON Point) already exists. No vectorIndex, no `route_posts_raw` table, no `route_matches` table.

**Desired state:**
- `curatedRouteValidator` has new `v.optional()` fields: `searchEmbedding`, `searchText`, `candidateIdentifiers`, `matchConfidence`, `llmReconciliationLog`, `description`, `rating`, `designation`, `sourceUrl`, `sourceRefs`, `highwayNumber`, `elevationGainM`, `surface`, `aadt`, `aadtMedian`, `aadtMax`, `pavementIri`, `mentionFrequency`, `mentionFrequencyScore`, `designationScore`, `elevationDramaScore`, `roadQualityScore`, `lowTrafficScore`, `weatherSuitability`, `bestMonths`, `sourceCount`, `qualityTier`.
- `curated_routes` table has `.vectorIndex("by_embedding", { vectorField: "searchEmbedding", dimensions: 1536 })`.
- New `route_posts_raw` table exists with `routePostRawValidator` and 3 indexes.
- New `route_matches` table exists with `routeMatchValidator` and 3 indexes.

**Architectural note:** The previously-planned `route_mentions` table is REPLACED by two tables: `route_posts_raw` (source of truth for raw LLM extractions) and `route_matches` (audit log of match decisions). This separation cleanly models the fact that one post can match zero, one, or many routes, and preserves the extraction artifact even if matching fails or is re-run with a different model.

- `by_postId` on `["postId"]`

Registering the vectorIndex and tables in `convex/schema.ts`:
```typescript
import { defineSchema, defineTable } from "convex/server";
import {
  curatedRouteValidator,
  routePostRawValidator,
  routeMatchValidator,
} from "./models/curated-routes";

export default defineSchema({
  curated_routes: defineTable(curatedRouteValidator)
    // ... existing indexes (DO NOT MODIFY)
    .index("by_source", ["source"])
    .index("by_state", ["state"])
    // NEW: vector index for semantic matching
    .vectorIndex("by_embedding", {
      vectorField: "searchEmbedding",
      dimensions: 1536,
      filterFields: ["state"],  // allow filtered vector search by state
    }),

  route_posts_raw: defineTable(routePostRawValidator)
    .index("by_postId", ["postId"])
    .index("by_source_and_extracted_at", ["source", "extractedAt"])
    .index("by_extraction_schema_version", ["extractionSchemaVersion"]),

  route_matches: defineTable(routeMatchValidator)
    .index("by_postId", ["postId"])
    .index("by_routeId", ["routeId"])
    .index("by_routeId_and_confidence", ["routeId", "matchConfidence"]),
});
```

Anti-pattern: Do NOT use `v.nullish()` — it allows both null and undefined, which can cause unexpected behavior in Convex validators. Always use `v.optional()`.

Anti-pattern: Do NOT store embeddings as JSON strings or as `v.record()` — use `v.array(v.number())` so Convex can index them.

- **The previously-planned route_mentions table is REPLACED by two tables:** `route_posts_raw` (raw artifacts) and `route_matches` (audit log). This cleanly separates "what did the LLM extract" from "which route did we match it to" — one post can match zero, one, or many routes, and we can re-run matching with a new model without losing the raw extraction.
- **vectorIndex dimensions is locked to 1536** to match OpenAI `text-embedding-3-small`. If we later switch to a different model with different dimensions (e.g., 3072 for text-embedding-3-large), we need a schema migration — treat this as a constraint.
- **filterFields: ["state"]** on the vectorIndex is valuable — it lets downstream code do `search(embedding, filter={state: "TN"})` for narrowed retrieval. Other filter fields (e.g., primary archetype) can be added later.
- **qualityTier values:** 'premium', 'standard', 'minimal' — Convex has no enum validators, so use v.string() and enforce the allowed set in application code.
- **sentimentLabel inside route_posts_raw.payload** is v.string() — values are 'positive', 'neutral', 'negative'.
- **llmReconciliationLog is an array of objects**, not a string or a record — this preserves structure and allows filtering by runId or reconciledAt.
- **mentionDate from the old plan is NOT reproduced** — it's replaced by postedAt in route_posts_raw.
- **Boy Scout rule:** If you spot a typo or missing index on an existing field while editing these files, fix it.
