# US-032: Connect saved routes list to new backend

> Task ID: US-032
> Type: FEATURE
> Priority: P0
> Estimate: 60 minutes
> Assignee: ui-developer

## CRITICAL CONSTRAINTS

### MUST
- Wire existing saved routes list to new Convex savedRoutes endpoints
- Real-time name search (filters as user types)
- Date range filter
- Clear filters button that resets all filters
- Empty state with CTA when no routes saved
- 'No results' state when filters match nothing

### NEVER
- Implement search on the backend — use client-side filtering on the loaded list
- Break existing list scroll behavior

### STRICTLY
- Search is case-insensitive substring match on route name
- Date filter compares route _creationTime against selected range
- Empty state CTA navigates to the chat/planning tab

## SPECIFICATION

**Objective:** Wire the existing saved routes list UI to the new Convex backend endpoints with search and filter capabilities.

**Success looks like:** Saved routes list loads from Convex, supports real-time name search, date filtering, and clear filters. Empty and no-results states are handled.

### Interaction Notes
- Search input at top of list with debounce (300ms)
- Date filter as a segmented control or date picker: Today, Last 7 days, Last 30 days, All
- Client-side filtering: load full list from Convex, filter locally
- Empty state reuses the generic empty-state component from Epic 2
- Pull-to-refresh support

### Filter Architecture
```
Convex list query (paginated, newest first)
  → Local state: searchQuery, dateRange
  → useMemo filter: name match + date range
  → FlatList renders filtered results
```

## NOTES

Client-side filtering is the right choice for MVP given the expected dataset size (dozens to low hundreds of routes per user). If the dataset grows, search can be moved server-side later. The date filter uses preset ranges rather than a custom date picker to keep the UI simple.
