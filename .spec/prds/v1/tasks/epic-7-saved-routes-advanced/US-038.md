## CRITICAL CONSTRAINTS

### MUST
- Detect 'include my favorite roads' in chat or session settings toggle
- Pass saved road segments as preferred corridors to route generation pipeline
- Show 'Includes your favorites' badge on applicable route cards
- Show 'couldn't include [Name] — too far' message for incompatible favorites

### NEVER
- Force favorite roads into routes where they are geographically incompatible
- Silently ignore the favorite roads request — always acknowledge it

### STRICTLY
- Favorite roads that add > 30% to total distance should trigger the "too far" message

## SPECIFICATION

**Objective:** Wire favorite saved road segments into the route planning pipeline so users can request their preferred roads be included in new route plans.

**Success looks like:** User says "include my favorite roads" in chat or toggles the setting, and the generated routes incorporate compatible favorites with clear badges, while incompatible ones get an explanatory message.

### Code Pattern

```typescript
// In orchestrator, after intent detection
if (includesFavorites) {
  const favoriteSegments = await ctx.runQuery(
    api.db.savedRoutes.getFavoriteSegments,
    { userId }
  )
  routeRequest.preferredCorridors = favoriteSegments.map(seg => ({
    name: seg.name,
    polyline: seg.polyline,
  }))
}

// After route generation, check which favorites were included
const included = checkFavoriteInclusion(generatedRoute, favoriteSegments)
const excluded = favoriteSegments.filter(s => !included.has(s.name))
```

### Anti-pattern (DO NOT)
Do NOT silently drop favorite roads — always report back which were included and which were too far. Do NOT modify the routing provider API — adapt at the orchestrator level.
