# US-025: Wire enrichRoute into planRideOrchestrator

> Task ID: US-025
> Type: FEATURE
> Priority: P1
> Estimate: 45 minutes
> Assignee: backend-engineer

## CRITICAL CONSTRAINTS

### MUST
- Call enrichRoute after conditions step in the orchestrator
- Single enrichRoute call for ALL routes (batch, not per-route)
- Apply labels, rationale, and highlights from enrichRoute response
- enrichRoute failure must NOT block the pipeline — fallback labels must exist
- Add highlights?: string[] to PlannedRouteOptionView

### NEVER
- Call enrichRoute per-route (must be a single batched call)
- Allow enrichRoute failure to prevent route delivery
- Skip fallback labels when enrichRoute fails

### STRICTLY
- enrichRoute is called after conditions, before final option assembly

## SPECIFICATION

**Objective:** Integrate the enrichRoute LLM tool into the planRideOrchestrator pipeline so that routes receive enriched labels, rationale, and highlights.

**Success looks like:** Routes have enriched labels and highlights from the LLM. When enrichRoute fails, routes still return with deterministic fallback labels.

## NOTES

enrichRoute is an LLM-powered tool, making it inherently probabilistic. The orchestrator must treat its output as best-effort. The fallback path (deterministic labels) must always work. This is a textbook application of the deterministic vs. probabilistic architecture pattern.
