# US-029: Implement departure time adjustment and weather error recovery

> Task ID: US-029
> Type: FEATURE
> Priority: P1
> Estimate: 60 minutes
> Assignee: backend-engineer

## CRITICAL CONSTRAINTS

### MUST
- Detect departure time in chat messages (e.g., 'what if I leave at 3pm')
- Extract time, re-fetch weather for new departure time, re-rank routes
- Agent notes weather unavailability in chat response
- Route cards show 'Weather unavailable' indicator when weather fails
- Staleness warning when weather data is >2 hours old
- Routes never blocked by weather failure

### NEVER
- Block route generation due to weather API failure
- Silently use stale weather data without warning
- Parse departure time with regex alone (use LLM extraction)

### STRICTLY
- Staleness threshold: 2 hours from weatherFetchedAt
- Weather unavailability communicated in chat AND on route cards

## SPECIFICATION

**Objective:** Enable riders to adjust departure time via chat and handle weather service failures gracefully so routes are always available.

**Success looks like:** Typing 'what if I leave at 3pm' triggers a re-fetch of weather data for the new departure time and re-ranks routes. When weather APIs fail, routes are still generated with clear 'unavailable' indicators. Stale weather data shows a warning.

### Departure Time Detection
- Agent uses LLM to detect departure time intent in user messages
- Extracted time is passed as departureTime parameter to planRide
- planRide re-fetches weather for the new time window

### Error Recovery Flow
```
Weather fetch fails
  → Log warning
  → Set overlays to undefined
  → Set weatherFetchedAt to null
  → Continue pipeline
  → Agent includes "Weather data unavailable" in response
  → Route cards show unavailable indicator (US-027)
```

## CODING STANDARDS

- Weather failure handling is deterministic — no LLM decides whether to skip weather
- Staleness check is a pure function comparing timestamps
- Error messages should be user-friendly, not technical

## NOTES

This task bridges the agentic chat layer (departure time detection via LLM) and the deterministic pipeline (weather re-fetch and re-ranking). The LLM extracts the time, but all subsequent logic (re-fetch, re-rank, error handling) is deterministic. This is a clean boundary between probabilistic and deterministic per the architecture guidelines.
