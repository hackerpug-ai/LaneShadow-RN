## CRITICAL CONSTRAINTS

### MUST
- Scroll position is preserved when navigating back from detail to list
- Scroll position is preserved when switching tabs and returning to Saved Routes
- Use React/Expo Router natural preservation (not manual scroll offset storage)

### NEVER
- Store scroll position in Convex or AsyncStorage
- Force scroll-to-top on every navigation back to list
- Manually save/restore scroll offset in useState or context

### STRICTLY
- Solution must work with the Expo Router Stack navigation pattern
- Let the framework handle scroll preservation naturally

## SPECIFICATION

**Objective:** Ensure the saved routes FlatList scroll position is preserved when the user navigates to a route detail and then returns to the list.

**Success looks like:** User scrolls down in saved routes list, taps a card, views detail, presses back - list is at the same scroll position they left.

## ACCEPTANCE CRITERIA

| # | Given | When | Then | Verify |
|---|-------|------|------|--------|
| 1 | User has scrolled to the 5th route card in a list of 10+ routes | User taps the 5th card, views detail, then navigates back | FlatList scroll position is at or near the 5th card (not scrolled to top) | `npx vitest run app/(app)/(tabs)/saved-routes.test.tsx` |
| 2 | User is at the bottom of the list | User taps last card, views detail, navigates back | List remains scrolled to the bottom position | `npx vitest run app/(app)/(tabs)/saved-routes.test.tsx` |
| 3 | User navigates to detail and a new route is added while on detail screen | User navigates back to list | List updates with new data but maintains approximate scroll position | `npx vitest run app/(app)/(tabs)/saved-routes.test.tsx` |
| 4 | User is on the saved routes list with scroll position | User switches to a different tab and back to Saved Routes tab | Scroll position is maintained (tab screens are kept mounted) | `npx vitest run app/(app)/(tabs)/saved-routes.test.tsx` |

## DESIGN

### References

- app/(app)/(tabs)/_layout.tsx (tab configuration)

### Interaction Notes
- Expo Router Stack navigation naturally preserves the tabs screen when pushing saved-route/[id]
- The FlatList in saved-routes.tsx should maintain scroll position if the component stays mounted
- Key insight: since saved-route/[id] is a Stack.Screen pushed on top of tabs, the tabs screen stays mounted
- This may be "free" with proper Expo Router stack navigation - verify first before adding workarounds
- If scroll resets due to unnecessary re-renders, investigate and fix root cause
- If still needed, use FlatList maintainVisibleContentPosition={{ minIndexForVisible: 0 }} prop
- Tab switching preserves state because Expo Router tabs keep screens mounted

### Code Pattern
Source: React Native FlatList API

```typescript
// Only add if natural preservation doesn't work:
<FlatList
  ref={flatListRef}
  maintainVisibleContentPosition={{ minIndexForVisible: 0 }}
  ...
/>
```

### Anti-pattern (DO NOT)
Do NOT manually save/restore scroll offset in useState or context. Do NOT use onScroll with scrollToOffset on mount. Let the framework handle it naturally. Do NOT add workarounds before verifying the default behavior.
