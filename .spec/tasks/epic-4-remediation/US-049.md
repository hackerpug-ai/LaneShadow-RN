# TASK: US-049 - Build "Couldn't Include" Message

**Task Type**: FEATURE
**Priority**: P0
**Estimate**: 75 minutes

## CRITICAL CONSTRAINTS

### MUST
- MUST display message when favorites are excluded due to distance
- MUST list names of excluded favorites in the message
- MUST only show message when at least one favorite is excluded
- MUST NOT show message when all favorites are included
- MUST be dismissible or auto-dismiss after timeout

### NEVER
- NEVER show message when includeFavorites toggle is OFF
- NEVER show message for network/permission errors (use different error UI)
- NEVER block route planning when favorites are excluded
- NEVER show message more than once per planning session

### STRICTLY
- STRICTLY follow app's info/alert message patterns
- STRICTLY use friendly, non-alarmist language
- STRICTLY ensure message is accessible (screen reader, dismissible)
- STRICTLY position message prominently but not obtrusively

## SPECIFICATION

**Objective**: Display an informative message to users when some of their favorite roads couldn't be included in the route because they're too far away (>50km from the route corridor).

**Success looks like**: When routes are generated and some favorites are excluded due to distance, a non-intrusive info message appears listing the excluded favorites by name, with clear messaging that they were too far from the planned route.

## ACCEPTANCE CRITERIA

| # | Given | When | Then | Verify |
|---|-------|------|------|--------|
| 1 | Routes generated with excluded favorites | Some favorites > 50km away | Info message appears listing excluded favorites | Visual inspection |
| 2 | Message displays | Favorites excluded | Message shows names of excluded favorites | Visual inspection |
| 3 | All favorites included | Routes generated | No exclusion message appears | Visual inspection |
| 4 | includeFavorites is OFF | User plans route | No exclusion message appears | Visual inspection |
| 5 | User taps message | Message is shown | Message dismisses on tap | Visual inspection |
| 6 | Message appears | 10 seconds elapsed | Message auto-dismisses | Visual inspection |
| 7 | Screen reader enabled | Message appears | Full message content is announced | Accessibility audit |
| 8 | Multiple routes with same exclusions | User regenerates routes | Message doesn't duplicate (session-aware) | Visual inspection |

## NOTES

**Component Location**:
- Create new component: FavoriteExclusionAlert or similar
- Render in route options container (above cards or below header)
- Look for existing info/alert patterns in the app

**Message Copy**:
- Title: "Some favorites couldn't be included"
- Body: "These favorites are too far from your route: {favorite1}, {favorite2}, {favorite3}"
- Optional: "Favorites must be within 50km of your route to be included."
- Tone: Informative, not apologetic or alarming

**Implementation Steps**:
1. Create FavoriteExclusionAlert component
2. Accept excludedFavorites array (with names)
3. Format favorite list (comma-separated, max 3, then "and N more")
4. Add dismiss button and auto-dismiss timer
5. Add accessibility labels and announcements
6. Integrate into route options container
7. Conditionally render based on excludedFavorites.length > 0

**Message Placement**:
- Above route option cards
- Below planning header/toggle
- Full-width or centered container
- Should not overlap or obstruct route cards

**Styling**:
- Use app's info/warning alert style (likely yellow/orange)
- Subtle, not alarming (this is expected behavior)
- Dismissible X icon in corner
- Smooth fade-in/fade-out animations

**Accessibility**:
- Role: "alert" or "status"
- AccessibilityLabel: Full message content
- Dismissible via tap and escape key (if platform supports)
- Focus management for keyboard users
- Test with VoiceOver/TalkBack

**Edge Cases**:
- No excluded favorites: Don't render message
- All favorites excluded: Still show message (this is valid)
- 10+ excluded favorites: Truncate list (show first 3 + "and 7 more")
- Duplicate exclusions (user regenerates): Show only once per session

**Session Management**:
- Track shown exclusions in session state
- Don't show same exclusion message twice
- Reset session when user changes planning parameters

**Testing**:
- Unit test for message rendering with different exclusion counts
- Integration test with route options container
- Snapshot test for message appearance
- Accessibility test with screen reader
- Manual test for auto-dismiss behavior

**Performance**:
- Message render should be instant
- Auto-dismiss timer should cleanup properly
- No memory leaks from timers or event listeners
