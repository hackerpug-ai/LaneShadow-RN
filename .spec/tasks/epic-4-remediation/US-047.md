# TASK: US-047 - Integrate Favorites with Planning Graph

**Task Type**: FEATURE + SPIKE
**Priority**: P0
**Estimate**: 240 minutes (includes 60min spike)

## CRITICAL CONSTRAINTS

### MUST
- MUST complete spike first to verify planning graph API capabilities
- MUST fetch user favorites via favoriteRoads.listByOwner when toggle enabled
- MUST filter favorites by proximity (50km threshold from route corridor)
- MUST pass filtered favorites to planning graph as preferred segments
- MUST return inclusion/exclusion lists for UI feedback

### NEVER
- NEVER bypass proximity filtering (must exclude distant favorites)
- NEVER block planning if favorites fetch fails (graceful degradation)
- EVER modify planning graph implementation directly (use existing API)
- NEVER include favorites when toggle is OFF

### STRICTLY
- STRICTLY follow existing planRide action patterns
- STRICTLY use existing distance calculation utilities (Haversine/similar)
- STRICTLY maintain route quality (favorites should be preference, not requirement)
- STRICTLY track which favorites were included for each route option

## SPECIFICATION

**Objective**: Modify the planRide backend action to accept favorite road geometries and use them as preferred segments in route planning.

**Success looks like**: When a user enables "Include favorite roads", the planning graph receives their nearby favorite road geometries and generates routes that prefer those segments when feasible, returning data about which favorites influenced each route.

**Implementation Approach**:
1. Add includeFavorites parameter to planRide schema
2. When true, fetch favorites with favoriteRoads.listByOwner
3. For each favorite, calculate distance from route start/end
4. Filter to favorites within 50km of route corridor
5. Pass filtered favorites to planning graph (API-dependent)
6. Return metadata: includedFavorites, excludedFavorites
7. Add error handling for favorites fetch failure

**Distance Calculation**:
- Use Haversine formula or existing utility
- Calculate from favorite bounds to route corridor
- 50km threshold is configurable via environment variable

**Testing**:
- Unit tests for distance calculation
- Integration tests for planRide with/without favorites
- Error handling tests for API failures
- Performance tests for large favorite sets (100+)

**Performance Considerations**:
- Favorites fetch should be async and non-blocking
- Consider pagination if user has 100+ favorites
- Cache favorites for 5-10 minutes during planning session

**Rollback Plan**:
- If planning graph doesn't support favorites, pass as waypoints instead
- If API changes are blocked, use post-processing to favor routes with favorites
