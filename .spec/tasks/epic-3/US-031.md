## CRITICAL CONSTRAINTS

### MUST
- Add `deletedAt` check to `softDeleteRouteHandler`: if route already has `deletedAt` set, return existing `scheduledDeletionId` (no-op, no double-schedule)
- Add `deletedAt` check to `permanentlyDeleteRouteHandler`: if route's `deletedAt` is undefined (undo happened), skip deletion
- Ensure both guards are tested with explicit race condition scenarios

### NEVER
- Change the 5-second undo window duration
- Remove the scheduler cancel in `undoDeleteRouteHandler`
- Change the public mutation signatures

### STRICTLY
- `softDeleteRouteHandler` must be idempotent: calling it twice on the same route must not create two scheduled deletions
- `permanentlyDeleteRouteHandler` must respect undo: if `deletedAt` is cleared, the route survives
- Both guards must be unit tested

## SPECIFICATION

**Objective:** Prevent data loss from soft-delete race conditions: double-invocation and undo-timing races.

**Success looks like:** Double-calling `softDeleteRoute` creates only one scheduled deletion. Undoing a delete at 4.9s while the scheduler fires at 5.0s does not permanently delete the route.

### Interaction Notes
- `softDeleteRouteHandler` guard: check `doc.deletedAt !== undefined` before scheduling. If already deleted, return `{ scheduledDeletionId: doc.scheduledDeletionId }`
- `permanentlyDeleteRouteHandler` guard: check `doc.deletedAt !== undefined` before deleting. If `deletedAt` is cleared, return null (undo won)
- Both are one-line guards with minimal risk

### Code Pattern
Source: convex/db/savedRoutes.ts:61-78

```typescript
// In softDeleteRouteHandler, after fetching doc:
if (doc.deletedAt !== undefined) {
  // Already soft-deleted, return existing scheduled ID (idempotent)
  return { scheduledDeletionId: doc.scheduledDeletionId! }
}

// In permanentlyDeleteRouteHandler, after fetching doc:
if (doc.deletedAt === undefined) {
  // Undo was called, route was restored - do not delete
  return null
}
```

### Anti-pattern (DO NOT)
Do NOT remove the scheduler.cancel() from undoDeleteRouteHandler -- keep both the cancel AND the guard as defense-in-depth. Do NOT add a lock/mutex pattern (Convex transactions handle atomicity).
