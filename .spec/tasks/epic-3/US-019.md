## CRITICAL CONSTRAINTS

### MUST
- Add optional `searchQuery` string arg to `getSavedRoutesList` and `listByOwner`
- Filter routes where `name` contains the search string (case-insensitive)
- Return the same response shape as the existing `getSavedRoutesList` query
- Maintain backward compatibility: omitting `searchQuery` returns all routes

### NEVER
- Add a new Convex search index (client-side filtering is sufficient for MVP scale)
- Change the existing return type validators
- Remove or rename existing query functions

### STRICTLY
- Case-insensitive matching via `toLowerCase()` comparison
- Empty string or undefined `searchQuery` must return all routes (no filter applied)
- Filter applied after fetching from DB index (post-fetch filter pattern)

## SPECIFICATION

**Objective:** Extend the existing `getSavedRoutesList` query to accept an optional search string and filter routes by name match.

**Success looks like:** Calling `getSavedRoutesList({ searchQuery: "morning" })` returns only routes whose name contains "morning" (case-insensitive). Omitting the arg returns all routes as before.
