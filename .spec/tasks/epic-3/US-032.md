## CRITICAL CONSTRAINTS

### MUST
- Remove `deleteRoute` public mutation from the API surface (or convert to internalMutation)
- Remove or update `useDeleteRoute` hook if it references the hard-delete path
- Keep `deleteById` as an internalMutation for potential admin/migration use
- Ensure all UI consumers use `softDeleteRoute` + `undoDeleteRoute` for the undo flow

### NEVER
- Remove `deleteById` entirely (keep as internal for programmatic use)
- Break existing `softDeleteRoute` / `undoDeleteRoute` flow
- Leave dead public mutations in the API

### STRICTLY
- After this change, no public mutation can permanently delete a route without the undo window
- The `useDeleteRoute` hook must either be removed or rewired to `softDeleteRoute`
- All tests referencing `deleteRoute` must be updated

## SPECIFICATION

**Objective:** Eliminate the public hard-delete bypass so all user-facing deletions go through the soft-delete-with-undo flow.

**Success looks like:** No client can call a public mutation that permanently deletes a route without the 5-second undo window. `deleteById` remains as an internalMutation for backend-only use.
