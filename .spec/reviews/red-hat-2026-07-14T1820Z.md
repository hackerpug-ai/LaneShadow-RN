## HIGH Confidence Findings

### H-1: Score-normalization × rider-ready threshold seam — `normalizeStates` will silently flip score-normalized rows to `riderReady=false`

**Severity:** HIGH
**Files:** `convex/curatedGeometry.ts:432` (predicate), `convex/curatedGeometryHygiene.ts:147-194` (normalizeEditorialScores), `convex/curatedGeometryHygiene.ts:768` (normalizeStates recompute hook)

**Evidence:**

The rider-ready predicate `computeRiderReadyFromDoc` gates on the 0-100 scale:

```ts
// convex/curatedGeometry.ts:429-438
return (
  gatePass &&
  doc.name.length > 0 &&
  doc.compositeScore >= 50 &&   // ← 0-100 scale threshold
  doc.lengthMiles > 0 &&
  rideWorthinessOk &&
  doc.retiredAt == null &&
  doc.duplicateOf == null &&
  doc.quarantine == null
)
```

S3-T1's `normalizeEditorialScores` divides compositeScore by 100, moving normalized
rows to the 0-1 scale (verified: tests assert `compositeScore: 0.85`, `0.9` at
`integration.test.ts:325,358`). But `normalizeEditorialScores` does **NOT** call
`recomputeRiderReadyForRoute` after patching — so the stored `riderReady` flag
remains stale (true) and the bug is dormant through the score pass.

The bug activates in the state-normalization pass. `normalizeStates`
(`curatedGeometryHygiene.ts:758-770`) calls `recomputeRiderReadyForRoute` on every
state-changed row:

```ts
// convex/curatedGeometryHygiene.ts:767-770
await ctx.db.patch(row._id, patch)
await ctx.runMutation(internal.curatedGeometry.recomputeRiderReadyForRoute, {
  id: row._id,
})
```

When `recomputeRiderReadyForRoute` runs on a row whose `compositeScore` was already
÷100 (e.g., 0.88), the predicate evaluates `0.88 >= 50` → **false** →
`riderReady = false`. The row silently loses rider-ready status.

**Production impact (real catalog):** The gate evidence shows 104 score-normalized
rows and 977 state-changed rows out of 5,757. Assuming uniform overlap, ~18
legitimate, gate-passing, high-quality editorial routes would silently flip to
`riderReady=false` when the human-gate pass sequence runs
(`normalizeEditorialScores` → `normalizeStates`). These rows would disappear from
the national-best `listCuratedRoutes` mode (which filters on `riderReady=true` via
the `by_riderReady_and_composite_score` index).

**Why the test suite missed it:** Every integration test runs each handler in
isolation with `routeIdPrefix` scoping. The score-normalization tests never run
state normalization on the same rows, and the rider-ready-exclusion test seeds
rows on the 0-100 scale (`compositeScore >= 50` passes). No integration test
exercises the full pass sequence (normalize scores → normalize states) on the same
fixture row and asserts `riderReady` stays true. The seam between S3-T1 and S3-T3
is completely uncovered.

**Note:** The quarantine handlers (`fixLengthOutliers:579`,
`quarantineTestRows:652`) also call `recomputeRiderReadyForRoute`, but for
quarantined rows `riderReady=false` is the correct outcome by coincidence
(quarantine check fails first), so the bug is masked there.

**Fix:** Either (a) update the rider-ready threshold to the 0-1 scale
(`compositeScore >= 0.5`) — but this is a cross-cutting change that must be
validated against every consumer of `riderReady`; OR (b) update
`normalizeEditorialScores` to call `recomputeRiderReadyForRoute` after patching
AND update the threshold; OR (c) at minimum, add an integration test that runs the
full pass sequence on the same fixture row and asserts `riderReady` is preserved,
then disposition the threshold as a separate explicit decision. The current state
— normalize scores without recomputing, leave the threshold on the wrong scale,
and let the next pass trigger the latent flip — is a correctness landmine.

### M-1: `findCandidateRoutesByEmbedding` does not filter `duplicateOf` shadows

**Severity:** MEDIUM
**Files:** `convex/semanticSearch.ts:51-107` (pure-vector query), used by `scripts/curation/pipeline/dedup/semantic_deduplicator.py:158`

**Evidence:** The commit `224e9b0f` ("close catalog shadow read-path gaps") closed
shadow exclusion in `findRoutesByIdentifier` (name, highway, candidateIdentifiers
paths) and `findCandidateRoutesHybrid` (same + vector-hit assembly). But the
**pure-vector** query `findCandidateRoutesByEmbedding` — which fetches documents by
`_id` after vector search (lines 88-102) — has no `duplicateOf` filter. Shadow
rows retain their `searchEmbedding` after dedupe (duplicateOf is a flag, not a
delete), so they remain in the vector index and surface as match candidates.

This is a live read path: `scripts/curation/pipeline/dedup/semantic_deduplicator.py:158`
calls it via HTTP, and Sprint 06 (QUAL-001) and Sprint 10 treat it as
production-ready. After the Founder commits the dedupe merge, this query would
return shadow rows alongside canonicals, polluting the curation dedup pipeline's
candidate set.

**Why MEDIUM not HIGH:** This is a curation-pipeline path, not a rider-facing
search path. The sprint's AC-4 specifically names `listCuratedRoutes` and the name
search. But the commit message claims to "close catalog shadow read-path gaps" —
this path remains open.

**Fix:** Add `if (doc.duplicateOf) return null` to the vector-hit fetch loop
(lines 89-101), consistent with the hybrid path's fix at line 761.

### M-3: `toHaveLength(1)` on the shadow-exclusion name search is fragile against real catalog data

**Severity:** MEDIUM
**Files:** `convex/__tests__/curatedGeometryHygiene.integration.test.ts:939-943`

**Evidence:** The new assertion:
```ts
const nameResult = parseResult(runConvexFn('semanticSearch:findRoutesByIdentifier', {
  identifier: 'Cherohala Skyway',
}))
expect(nameResult).toHaveLength(1)
expect(nameResult[0].routeId).toBe(canonicalId)
```

`findRoutesByIdentifier('Cherohala Skyway')` searches globally by `name_lower` —
no `stateFilter`, no prefix scope. The test's `dedupeGroups` call is scoped to
`routeIdPrefix: 'test:cherohala-'`, so only the fixture rows are deduped. But the
real dev catalog **also** has Cherohala Skyway rows (gate evidence:
"cherohala skyway → canonical=cherohala-skyway, 3 shadows"), and those real rows
are NOT yet deduped (founder gate pending → `duplicateOf` is null). The global name
search would return the test canonical + any real Cherohala rows, making the count
>1 and the `toHaveLength(1)` assertion fail.

The candidateIdentifiers path mitigates this with `stateFilter: 'Test Shadow State'`
(a unique fixture state), which correctly isolates the assertion. But the name path
has no such isolation. Because the evidence is stale (M-2), this potential failure
is unverified.

**Fix:** Either (a) assert `nameResult` contains the canonical and excludes the
shadows without asserting an exact global count, or (b) use a fixture name that
does not collide with real catalog data.
