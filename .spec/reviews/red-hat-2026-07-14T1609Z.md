## MEDIUM Confidence Findings

### M-1: Production-mode `normalizeEditorialScores` misses mixed-scale rows

**Severity:** MEDIUM
**Files:** `convex/curatedGeometryHygiene.ts:167-170`

**Evidence:** In production mode (no `routeIdPrefix`), the handler uses:
```ts
.withIndex('by_composite_score', (q) => q.gt('compositeScore', 1))
```
This fetches ONLY rows where `compositeScore > 1`. A mixed-scale row (compositeScore=0.85, curvatureScore=88) has compositeScore ≤ 1 and is **never fetched**. The `computeNormalizedScores` mixed-scale guard (REDHAT-FIX-001) correctly handles these rows via `SCORE_FIELDS.some(k => row[k] > 1)`, but the guard is **unreachable in production** because the index pre-filters them out.

The REDHAT-FIX-001 test exercises mixed-scale only through `routeIdPrefix:'test:hyg-mixed-'` (broad `by_routeId` scan mode). The task spec explicitly accepted this tradeoff ("when no prefix (production), keep the by_composite_score index scan for read efficiency"), but it leaves a real gap: if any production row has a partially-normalized state (compositeScore already ÷100 but dimensions still on 0–100), those dimensions will never be corrected.

**Risk assessment:** Low for greenfield catalogs (all editorial scores start on 0–100, so compositeScore > 1 catches them all). Higher risk if a prior partial-normalization bug created mixed-scale rows. The dry-run found 104 out-of-scale rows — all caught by the index. No evidence of mixed-scale rows in the real catalog.

**Fix:** Either (a) add a one-time broad scan pass (no index filter) to catch any residual mixed-scale rows, or (b) document that production mode assumes no prior partial normalization.

### M-2: S3-T3 AC-1 "claimed length nulled" requirement is unmet

**Severity:** MEDIUM
**Files:** `convex/curatedGeometryHygiene.ts:575-578`

**Evidence:** The S3-T3 task spec AC-1 requires:
> "the ≤0 row is flagged quarantine.reason='zero_length' **with its claimed length nulled**"

The implementation patches `quarantine: { reason, flaggedAt }` but does NOT null `lengthMiles`. The test at `integration.test.ts:970-973` checks `quarantine.reason == 'zero_length'` but does NOT verify lengthMiles is nulled.

The `lengthMiles` field is `v.number()` (required, non-optional) in the schema — it cannot be set to `null` or `undefined` via a patch without a schema change. The spec also says "Never edit convex/schema.ts". This is a **spec contradiction**: the spec demands an operation the schema forbids.

**Impact:** Zero-length rows retain `lengthMiles=0` after quarantine. Downstream consumers that read `lengthMiles` directly (without checking quarantine) will see a length of 0 rather than null/absent. The quarantine flag is the correct exclusion signal, but the "nulled" semantics provide defense-in-depth that is missing.

**Fix:** Either (a) change `lengthMiles` to `v.optional(v.number())` in the schema and null it, or (b) formally descope this requirement with a note that quarantine is the sole exclusion signal.

**Severity:** LOW
**Files:** `convex/curatedGeometryHygiene.ts:491-498`

The internalAction batches shadow writes via `applyDedupeShadows` (50 per batch). If the action crashes after batch 1 but before batch 2, batch 1's shadows are committed (separate transactions). Mitigated by idempotency: re-running the action skips already-shadowed rows and processes the rest. Not atomic, but recoverable.

The `DedupeRow` type uses `_id: any` instead of `Id<'curated_routes'>`. The `applyDedupeShadows` mutation validator (`v.id('curated_routes')`) provides runtime safety, but the TS type is loose. A refactor could silently pass a wrong ID type.

### L-5: `normalizeStates` does not clear stale `statesAll` on re-edit

## dedupeGroups internalAction / paginated-query / batched-mutation design assessment

**Architecture:** The dedupeGroups handler was converted from `internalMutation` (which crashed with `Too many bytes read in a single function execution (limit: 16777216 bytes)` on the real catalog) to `internalAction` at commit `12392fe8`. The new design:

1. **`fetchDedupePage`** (internalQuery): paginates reads via `.paginate({cursor, numItems})` with projected fields (9 fields only) to minimize bytes per page. Default page size 500. ✓
2. **`computeDedupePlan`** (pure function): groups by name_lower (fallback `name.toLowerCase()` for real catalog rows lacking name_lower), clusters via union-find with haversine ≤100mi, selects canonical (prefer `geometryStatus='generated'`, then highest score). ✓
3. **`applyDedupeShadows`** (internalMutation): batch-writes duplicateOf flags in batches of ≤50. ✓
4. **`dedupeGroups`** (internalAction): orchestrates the loop — paginate all reads into memory, compute plan, batch the writes.

**Correctness:** The algorithm is sound. Grouping by name + centroid proximity prevents over-merge. Canonical selection prefers gate-passing geometry. Shadows are reversible (duplicateOf string, not deletion). Idempotency holds (rows with duplicateOf set are skipped on re-run).

**Real-deployment behavior:** The dry-run on the actual 5,757-row catalog found 44 groups with 49 shadows — a plausible result for a multi-source aggregated catalog. The previous internalMutation version crashed; the internalAction version succeeds. The fix is real and necessary.

**Concerns:**
- The action loads ALL 5,757 rows into memory (~1-2MB projected) before computing the plan. This works for the current catalog size but won't scale to 50K+ rows without streaming grouping.
- The `_id: any` type (L-4) is a minor type-safety gap.
- Partial-write risk on crash (L-3) is mitigated by idempotency.
- The `name.toLowerCase()` fallback for rows without `name_lower` means the `by_name_lower` index is NOT used for grouping — the action does a full-table scan via pagination. This is correct but means grouping happens in memory, not via index. ✓ for correctness, ⚠ for scalability.
